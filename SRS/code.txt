\section{Scope}

\subsection{Product Name and Context}
The system to be developed is the \textbf{Upwork Proposal Assistant}, a specialized browser extension designed to optimize the workflow of professional freelancers operating on the Upwork platform. In the current competitive landscape of online freelancing, professionals spend significant time analyzing potential clients, crafting customized proposals, and managing their application history.

\subsection{Problem Statement}
Current freelance workflow challenges include:

\begin{itemize}
\item \textbf{Time Inefficiency:} Freelancers spend an average of 15-20 minutes per job application analyzing client history, reading reviews, and writing customized proposals.

\item \textbf{Information Overload:} Upwork job listings contain extensive data (client spending history, hire rates, previous feedback, job requirements) that must be manually processed and evaluated.

\item \textbf{Inconsistent Quality:} Without systematic analysis tools, freelancers may submit proposals to low-quality clients or fail to identify red flags in client behavior patterns.

\item \textbf{Resource Wastage:} Upwork's "Connect" system charges freelancers for each application. Applying to clients with poor hiring records wastes both Connects and time.

\item \textbf{Template Management:} Successful freelancers maintain libraries of proposal templates but lack tools to track which templates perform best or systematically improve their approach.
\end{itemize}

\subsection{Solution Overview}
The Upwork Proposal Assistant addresses these challenges through intelligent automation and data-driven decision support. The system will reduce proposal preparation time by approximately 60\% through the following core capabilities:

\subsubsection{Automated Client Analysis}
The system shall automatically extract and analyze client profile data including:
\begin{itemize}
\item Total spending history and budget patterns
\item Historical hire rate percentages
\item Previous freelancer feedback and sentiment analysis
\item Payment verification status and geographic location
\item Job posting patterns and response rates
\end{itemize}

This analysis will be synthesized into a normalized "Client Reliability Score" (1-100 scale) displayed prominently within the Upwork interface, enabling freelancers to make informed decisions about where to invest their limited Connects.

\subsubsection{AI-Powered Proposal Generation}
Leveraging large language models (LLMs), the system shall generate customized proposal drafts by:
\begin{itemize}
\item Parsing job description requirements and matching them to the freelancer's skill profile
\item Adapting tone and style based on client characteristics (corporate vs. startup, technical vs. non-technical)
\item Incorporating relevant portfolio examples and past project successes
\item Maintaining professional quality while reducing writing time from 15 minutes to under 2 minutes
\end{itemize}

\subsubsection{Template Management and Analytics}
The system shall provide sophisticated template management including:
\begin{itemize}
\item Storage and categorization of proposal templates
\item Performance tracking (views, interviews, hires per template)
\item Automated ranking based on success metrics
\item Historical analytics showing proposal effectiveness trends over time
\end{itemize}

\subsection{System Boundaries}
To ensure compliance with Upwork's Terms of Service and maintain ethical operation, the system explicitly \textbf{will NOT}:

\begin{itemize}
\item Automatically submit proposals without explicit user confirmation
\item Circumvent Upwork's Connect payment system
\item Violate Upwork's rate limiting or anti-automation policies
\item Store or transmit Upwork user credentials (authentication handled via OAuth where possible)
\item Scrape data from Upwork for purposes other than assisting the authenticated user
\item Share or aggregate data across multiple users without explicit consent
\end{itemize}

The system functions strictly as an "Assistant" – augmenting human decision-making rather than replacing it. All final decisions regarding proposal submission remain with the freelancer.

\subsection{Target Metrics and Success Criteria}
Upon successful implementation, the system shall achieve:

\begin{itemize}
\item \textbf{Time Reduction:} Average proposal preparation time reduced from 15 minutes to 6 minutes or less (60\% improvement)
\item \textbf{Decision Quality:} 80\% of users report improved ability to identify high-quality clients
\item \textbf{Resource Efficiency:} 50\% reduction in wasted Connects on low-probability applications
\item \textbf{User Satisfaction:} System Usability Scale (SUS) score of 75 or higher
\item \textbf{Performance:} 99.5\% of analysis operations completed within 3 seconds
\end{itemize}

\subsection{Benefits and Objectives}
The successful deployment of this system will provide:

\begin{enumerate}
\item \textbf{Competitive Advantage:} Freelancers can respond to job postings faster and more effectively than competitors using manual processes.

\item \textbf{Income Optimization:} By avoiding low-quality clients and targeting high-value opportunities, freelancers can increase their effective hourly rate.

\item \textbf{Professional Development:} Analytics and performance tracking enable freelancers to systematically improve their proposal strategies over time.

\item \textbf{Stress Reduction:} Automation of tedious tasks reduces cognitive load and allows freelancers to focus on high-value creative work.

\item \textbf{Scalability:} The system enables freelancers to efficiently manage higher volumes of applications without proportional time investment.
\end{enumerate}

\section{Definitions, Acronyms, and Abbreviations}

\begin{description}[leftmargin=4cm,style=nextline]

\item[API] Application Programming Interface: A set of protocols and tools that enables communication between different software components. In this system, APIs facilitate interaction between the browser extension, backend server, and external AI services.

\item[Client Reliability Score] A normalized numerical value (1-100) calculated by the system to represent the overall trustworthiness and quality of an Upwork client based on historical data analysis.

\item[Connect] Upwork's internal currency used by freelancers to submit proposals. Each job application costs a specific number of Connects (typically 1-6), and freelancers must purchase additional Connects when their allocation is exhausted.

\item[CSS Selector] A pattern-matching syntax used to identify and extract specific HTML elements from web pages. The system uses CSS selectors to scrape client data from Upwork's Document Object Model.

\item[DOM] Document Object Model: The hierarchical tree structure representing a web page's content. The browser extension manipulates the DOM to inject UI elements and extract job data.

\item[Extension Manifest] A JSON configuration file (manifest.json) that defines a browser extension's permissions, capabilities, scripts, and metadata. This system uses Manifest V3 specification.

\item[Freelancer] The primary end-user of this system; a professional service provider who seeks and completes projects through the Upwork platform.

\item[JSR] Job Success Rate: An Upwork metric representing the percentage of contracts a freelancer completes successfully, weighted by project value and client feedback.

\item[LLM] Large Language Model: An artificial intelligence system trained on vast text corpora capable of generating human-like text. This system uses LLMs (such as GPT-4 or Claude) for proposal generation.

\item[NFR] Non-Functional Requirement: A specification of system quality attributes (performance, security, usability) rather than specific behaviors or functions.

\item[OAuth] Open Authorization: An industry-standard protocol for secure delegated access, potentially used for Upwork API authentication if direct integration becomes available.

\item[Proposal] A formal written application submitted by a freelancer to a client's job posting, typically including an introduction, relevant experience, project approach, pricing, and call-to-action.

\item[RP] Requirement Provider: The individual or entity responsible for defining system goals, validating requirements, and accepting final deliverables. In academic context, this may be a faculty advisor or external client.

\item[REST API] Representational State Transfer Application Programming Interface: An architectural style for web services using HTTP methods (GET, POST, PUT, DELETE) for data exchange between the extension and backend server.

\item[SRS] Software Requirements Specification: This formal document describing all functional and non-functional requirements for the system being developed, conforming to IEEE Std 830-1984 guidelines.

\item[TBD] To Be Determined: A placeholder indicating that specific requirement details are not yet finalized and require future resolution by a specified milestone or meeting.

\item[TLS] Transport Layer Security: A cryptographic protocol ensuring secure, encrypted communication between the browser extension and backend servers, protecting user data during transmission.

\item[Upwork Platform] The online marketplace connecting freelancers with clients seeking professional services. The system interfaces with Upwork's web application but operates independently.

\item[Use Case] A specific scenario describing how an actor (user or external system) interacts with the system to achieve a particular goal, including preconditions, main flow, and expected outcomes.

\end{description}

\section{Product Perspective}

\subsection{System Context}
The Upwork Proposal Assistant exists within a complex ecosystem of interconnected technologies and platforms. Understanding this context is essential for proper system design and integration planning.

The system is neither a standalone product nor a component of a larger pre-existing software suite. Instead, it functions as an \textbf{augmentation layer} that integrates with existing platforms and services. The primary integration point is the Upwork web platform, accessed through standard web browsers (Chrome, Firefox, Edge, Brave). The system does not replace or fundamentally alter Upwork's functionality; rather, it enhances the user experience by providing additional analytical and automation capabilities.

\subsection{Architectural Overview}
The Upwork Proposal Assistant employs a three-tier distributed architecture:

\subsubsection{Tier 1: Client-Side Browser Extension}
The front-end component runs entirely within the user's web browser as a packaged extension conforming to Manifest V3 specifications. This component is responsible for:

\begin{itemize}
\item \textbf{DOM Manipulation:} Injecting visual overlays, badges, and sidebars into Upwork pages without disrupting native functionality
\item \textbf{Data Extraction:} Parsing HTML structure using CSS selectors and XPath queries to extract client data, job descriptions, and user profile information
\item \textbf{Local Storage Management:} Caching frequently accessed data (templates, user preferences) in browser local storage for offline access
\item \textbf{Event Handling:} Listening for user interactions (button clicks, page navigation) and triggering appropriate backend requests
\item \textbf{Real-Time UI Updates:} Displaying analysis results, proposal drafts, and notifications without requiring page refreshes
\end{itemize}

The extension maintains minimal footprint (under 5MB packaged size) to ensure fast installation and updates. All computationally intensive operations (AI inference, sentiment analysis) are delegated to the backend tier to prevent browser performance degradation.

\subsubsection{Tier 2: Backend Application Server}
A Node.js-based REST API server deployed on cloud infrastructure (AWS, Google Cloud, or Azure) handles business logic and orchestrates external service integrations. Core responsibilities include:

\begin{itemize}
\item \textbf{Authentication and Authorization:} Validating user sessions using JWT (JSON Web Tokens) and enforcing role-based access control
\item \textbf{Data Processing Pipeline:} Executing client analysis algorithms, sentiment scoring, and reliability calculations
\item \textbf{AI Service Integration:} Managing API calls to LLM providers (OpenAI, Anthropic) with request queuing, retry logic, and error handling
\item \textbf{Analytics Aggregation:} Processing proposal history data to generate performance reports and success metrics
\item \textbf{Cache Management:} Implementing Redis-based caching to reduce database load and improve response times
\item \textbf{Rate Limiting:} Enforcing usage quotas to prevent abuse and manage API costs
\end{itemize}

The backend architecture follows microservices principles, with distinct modules for analysis, generation, authentication, and analytics. This modularity enables independent scaling, testing, and deployment of individual components.

\subsubsection{Tier 3: Data Persistence Layer}
A MongoDB database cluster provides persistent storage for:

\begin{itemize}
\item \textbf{User Profiles:} Account information, skill configurations, preferences, and subscription tiers
\item \textbf{Proposal Templates:} User-created and system-provided templates with associated metadata (success rates, usage counts, tags)
\item \textbf{Historical Data:} Complete archive of generated proposals, client analyses, and decision outcomes for analytics
\item \textbf{System Configuration:} API keys, feature flags, rate limits, and operational parameters
\end{itemize}

MongoDB was selected for its flexible schema design, horizontal scalability, and native JSON document storage aligning with JavaScript-based application architecture.

\subsection{External System Dependencies}
The Upwork Proposal Assistant interfaces with several external systems:

\subsubsection{Upwork Web Platform (Primary External System)}
\begin{itemize}
\item \textbf{Interface Type:} Indirect DOM-based interaction (no official API integration)
\item \textbf{Data Flow:} Unidirectional read-only access to public job posting data
\item \textbf{Dependency Risk:} High – Changes to Upwork's HTML structure may break data extraction
\item \textbf{Mitigation Strategy:} Implement adaptive CSS selectors with fallback patterns and automated regression testing
\end{itemize}

\subsubsection{Large Language Model APIs (AI Services)}
\begin{itemize}
\item \textbf{Primary Provider:} OpenAI GPT-4 or Anthropic Claude (configurable)
\item \textbf{Interface Type:} RESTful HTTPS API using JSON payloads
\item \textbf{Data Flow:} Bidirectional – System sends context prompts; AI returns generated text
\item \textbf{Dependency Risk:} Medium – Service outages impact proposal generation but not analysis features
\item \textbf{Mitigation Strategy:} Implement provider failover and graceful degradation to manual templates
\end{itemize}

\subsubsection{Browser Extension Stores}
\begin{itemize}
\item \textbf{Platforms:} Chrome Web Store, Firefox Add-ons, Microsoft Edge Add-ons
\item \textbf{Interface Type:} Manual upload and automated update distribution
\item \textbf{Dependency Risk:} Low – Only affects distribution, not operation
\item \textbf{Compliance Requirements:} Must adhere to each platform's content policies and technical requirements
\end{itemize}

\subsection{System Boundaries and Constraints}

\subsubsection{Technical Boundaries}
The system operates exclusively within the boundaries of:
\begin{itemize}
\item Modern web browsers supporting JavaScript ES6+ and Web Extensions API
\item Upwork's publicly accessible job posting pages (does not access authenticated areas beyond user's own account)
\item HTTPS-encrypted communication channels for all external data transmission
\item User-granted browser permissions (no unauthorized data access)
\end{itemize}

\subsubsection{Functional Boundaries}
The system explicitly \textbf{does not}:
\begin{itemize}
\item Guarantee hiring success (provides decision support only)
\item Automate bid placement or contract acceptance
\item Modify Upwork's native functionality or appearance permanently
\item Store Upwork account credentials (uses session-based authentication)
\item Share individual user data across accounts without explicit consent
\end{itemize}

\subsubsection{Legal and Ethical Boundaries}
The system respects:
\begin{itemize}
\item Upwork's Terms of Service regarding automated access
\item GDPR and data protection regulations for user privacy
\item Intellectual property rights (no plagiarism in AI-generated content)
\item Fair use principles in data analysis and processing
\end{itemize}

\subsection{Operational Perspective}

\subsubsection{Deployment Model}
The system follows a hybrid deployment approach:

\begin{itemize}
\item \textbf{Extension:} Distributed through official browser extension marketplaces with automatic update mechanisms
\item \textbf{Backend:} Deployed on containerized cloud infrastructure (Docker/Kubernetes) with auto-scaling capabilities
\item \textbf{Database:} Managed MongoDB Atlas cluster with automated backups and point-in-time recovery
\end{itemize}

\subsubsection{Operational Lifecycle}
The typical user journey spans:
\begin{enumerate}
\item \textbf{Discovery and Installation:} User finds extension in browser store, reviews permissions, and installs
\item \textbf{Onboarding:} First-run configuration wizard collects profile data and preferences
\item \textbf{Daily Usage:} User browses Upwork jobs; system provides real-time analysis and proposal assistance
\item \textbf{Continuous Improvement:} System learns from user feedback and proposal outcomes to refine recommendations
\item \textbf{Subscription Management:} User may upgrade tiers for advanced features (premium AI models, unlimited analyses)
\end{enumerate}

\subsubsection{Maintenance and Support}
System maintenance encompasses:
\begin{itemize}
\item \textbf{Proactive Monitoring:} 24/7 automated health checks on API endpoints, database connections, and third-party service availability
\item \textbf{Reactive Support:} User-reported issues triaged through in-app feedback system and support ticketing
\item \textbf{Continuous Updates:} Monthly feature releases and weekly security patches delivered via automatic extension updates
\item \textbf{Data Integrity:} Daily automated backups with 30-day retention and quarterly disaster recovery drills
\end{itemize}

\section{Product Functions}

This section provides a high-level summary of system capabilities organized by functional area. Detailed specifications for each function appear in Chapter 4 (Functional Requirements).

\subsection{Client Intelligence and Analysis}

\subsubsection{Automated Profile Evaluation (FR-001)}
The system shall autonomously extract and analyze client historical data from Upwork job postings including total spending, hire rate percentages, payment verification status, and geographic location. This data shall be normalized into a Client Reliability Score (1-100 scale) using weighted algorithms that prioritize hiring history (40\%), spending patterns (30\%), and review sentiment (30\%).

\subsubsection{Sentiment Analysis of Client Reviews (FR-001)}
The system shall perform natural language processing on previous freelancer feedback to identify patterns indicating positive or negative client experiences. Red-flag keywords (e.g., "unresponsive," "payment issues," "scope creep") shall be automatically detected and factored into reliability calculations.

\subsubsection{Competitive Analysis (FR-003)}
The system shall evaluate the number of existing applicants, average interview rates, and job posting recency to estimate the probability of the user's proposal being viewed and considered. This competitive intelligence enables strategic decision-making about Connect allocation.

\subsection{Intelligent Proposal Generation}

\subsubsection{AI-Driven Draft Creation (FR-002)}
Leveraging large language models, the system shall generate customized proposal drafts by synthesizing job description requirements with the user's professional profile. The generated text shall maintain professional quality, incorporate relevant experience examples, and adapt tone based on client characteristics.

\subsubsection{Context-Aware Customization (FR-002, FR-007)}
The system shall analyze job descriptions to identify key requirements, preferred skills, and client priorities. Proposal generation shall emphasize matching qualifications and de-emphasize irrelevant experience, maximizing perceived fit.

\subsubsection{Style and Tone Adaptation (FR-007)}
Users shall configure proposal voice settings (Professional, Friendly, Technical, Concise) that influence AI generation parameters. The system shall maintain stylistic consistency while ensuring authenticity and avoiding generic template language.

\subsection{Template and Content Management}

\subsubsection{Template Library Management (FR-004)}
The system shall provide a comprehensive repository for storing, organizing, and retrieving proposal templates. Each template shall support dynamic placeholders (e.g., [Client\_Name], [Project\_Budget]) that automatically populate from job posting data.

\subsubsection{Performance-Based Ranking (FR-004)}
Templates shall be automatically ranked using a "Success Velocity" metric calculated as: $V = \frac{\text{Interviews}}{\text{Total Uses}} \times 100$. High-performing templates shall be promoted in selection interfaces, while consistently underperforming templates shall be flagged for revision.

\subsubsection{Template Optimization Recommendations (FR-004)}
The system shall analyze successful vs. unsuccessful proposals to identify effective keywords, optimal length ranges, and structural patterns. Users shall receive actionable suggestions for improving template performance.

\subsection{Historical Analytics and Performance Tracking}

\subsubsection{Comprehensive Proposal Archive (FR-005)}
The system shall maintain a complete historical record of all generated proposals, associated client analyses, and application outcomes (views, interviews, hires). This archive enables longitudinal performance analysis and strategy refinement.
\subsubsection{Success Metrics Dashboard (FR-005)}
Users shall access visual analytics including:
\begin{itemize}
\item Weekly proposal volume trends (line graphs)
\item Client reliability distribution of applied jobs (histogram)
\item Template performance comparison (bar charts)
\item Interview-to-application conversion rates (pie charts)
\item Average Connect cost per interview (calculated metric)
\end{itemize}
\subsubsection{Trend Analysis and Insights (FR-005)}
The system shall identify patterns such as:
\begin{itemize}
\item Time-of-day correlations with proposal success
\item Client characteristics associated with higher interview rates
\item Keyword effectiveness in different job categories
\item Seasonal variations in market competitiveness
\end{itemize}
\subsection{User Profile and Skill Configuration}
\subsubsection{Professional Identity Management (FR-006)}
Users shall define their professional profile including:
\begin{itemize}
\item Biographical summary and career highlights
\item Technical skill inventory with proficiency levels
\item Portfolio links and work sample references
\item Hourly rate ranges and availability preferences
\item Specialization areas and target industries
\end{itemize}
\subsubsection{Skill-Job Matching (FR-006)}
The system shall cross-reference job requirements against the user's configured skills to calculate a compatibility percentage. Applications to jobs where the user lacks 50%+ of required skills shall trigger warning notifications.
\subsubsection{Dynamic Profile Updates (FR-006)}
As users gain experience and complete projects, the system shall suggest profile updates to reflect new capabilities and accomplishments, ensuring proposal generation utilizes current information.
\subsection{Decision Support and Recommendations}
\subsubsection{Multi-Tier Analysis Engine (FR-003)}
The system shall apply sophisticated logic gates to generate application recommendations:
\begin{itemize}
\item \textbf{Tier 1 (Resource Protection):} Flag jobs with 50+ applicants and client hire rate below 30% as "High Risk - Consider Skipping"
\item \textbf{Tier 2 (Quality Scoring):} Categorize jobs as Strong Apply (score 80+), Caution (50-79), or Skip (below 50)
\item \textbf{Tier 3 (Urgency Detection):} Identify time-sensitive opportunities (posted within 30 minutes, high hire rate) with "Urgent Action" badges
\end{itemize}
\subsubsection{Reasoning Transparency (FR-003)}
Each recommendation shall include a plain-language explanation of the underlying rationale (e.g., "Client has 90% hire rate but pays 20% below market average for this skill level"), enabling users to understand and validate system logic.
\subsection{System Administration and Maintenance}
\subsubsection{Global Template Repository (FR-008)}
Administrators shall manage a curated library of high-performing templates shared across the user base (with appropriate anonymization). Community-sourced templates undergo quality review before publication.
\subsubsection{System Health Monitoring (FR-008)}
Administrative dashboards shall display real-time metrics including:
\begin{itemize}
\item API response time percentiles (p50, p95, p99)
\item Error rates by endpoint and failure type
\item Database query performance and connection pool utilization
\item Third-party service availability (AI APIs, external dependencies)
\end{itemize}
\subsubsection{Configuration Management (FR-008)}
Administrators shall control system-wide settings including:
\begin{itemize}
\item AI model selection and prompt engineering parameters
\item Rate limiting thresholds and usage quotas per tier
\item Feature flags for gradual rollout of new capabilities
\item API key rotation for security compliance
\end{itemize}
\subsection{Data Synchronization and Persistence}
\subsubsection{Cross-Device Sync (FR-009)}
User data (profiles, templates, settings) shall synchronize across multiple devices in near-real-time using event-driven architecture. Conflict resolution shall prioritize most-recent-write with user override options.
\subsubsection{Offline Capability (FR-009)}
The system shall cache essential data locally, enabling limited functionality (template access, manual proposal drafting) during network outages. Upon reconnection, pending operations shall queue for automatic synchronization.
\subsubsection{Data Security and Encryption (FR-009)}
All data transmission shall employ TLS 1.3 encryption. Data at rest shall be encrypted using AES-256. User-specific encryption keys shall derive from account credentials to ensure only authorized access.

\section{Operating Environment}
The Upwork Proposal Assistant must function reliably across diverse computing environments while maintaining consistent performance and user experience. This section defines the technical platforms, configurations, and environmental constraints within which the system operates.
\subsection{Client-Side Environment Requirements}
\subsubsection{Supported Web Browsers}
The browser extension shall be compatible with:
\begin{itemize}
\item \textbf{Google Chrome:} Version 100 and later (released April 2022+)
\item \textbf{Microsoft Edge:} Version 100 and later (Chromium-based versions only)
\item \textbf{Brave Browser:} Version 1.40 and later
\item \textbf{Mozilla Firefox:} Version 95 and later (released December 2021+)
\item \textbf{Opera:} Version 85 and later (Chromium-based)
\end{itemize}
\textbf{Rationale:} These minimum versions ensure support for Manifest V3, ES2020 JavaScript features, and modern Web Extensions APIs required for DOM manipulation and secure communication.
\textbf{Browser Limitations:} Safari is explicitly not supported in initial release due to incompatibilities between Safari's extension architecture and Chrome/Firefox standards. The system shall operate on:
\begin{itemize}
\item \textbf{Windows:} Windows 10 (Version 1909+) and Windows 11
\item \textbf{macOS:} macOS 12 (Monterey) and later versions
\item \textbf{Linux:} Ubuntu 20.04 LTS+, Fedora 35+, or any distribution supporting modern Chromium/Firefox builds
\end{itemize}
The system is operating system agnostic at the application level (no OS-specific APIs used), but OS choice affects browser availability and update mechanisms.

\section{FR-001: Automated Client Data Extraction}
\subsection{Description}
The system shall programmatically scrape and parse the Document Object Model (DOM) of the active Upwork job page to extract client-related metadata without manual user intervention.

\subsection{Inputs}
\begin{itemize}
    \item \textbf{Active Tab Context:} The URL of the current Upwork job post.
    \item \textbf{DOM Tree:} The raw HTML structure of the page.
\end{itemize}

\subsection{Processing}
\begin{enumerate}
    \item The system shall identify specific CSS selectors associated with client metrics.
    \item It shall parse the "Hire Rate," "Total Spent," and "Average Hourly Rate."
    \item The system shall extract the last 10 feedback entries to identify recurring sentiment patterns.
\end{enumerate}

\subsection{Outputs}
\begin{itemize}
    \item \textbf{Data Object:} A structured JSON object containing parsed metrics.
    \item \textbf{Status Event:} A trigger sent to the UI to update the loading state.
\end{itemize}

\section{FR-002: Client Reliability Scoring Engine}
\subsection{Description}
This module processes extracted data to generate a normalized 1-100 score representing the client's quality and trustworthiness.

\subsection{Inputs}
\begin{itemize}
    \item Client metadata from FR-001.
    \item Admin-defined weightage constants ($w_1, w_2, w_3$).
\end{itemize}

\subsection{Processing}
The system shall calculate the Final Score ($S$) using a weighted linear combination:
$$S = (w_1 \cdot \text{HireRate}) + (w_2 \cdot \text{SpendMetric}) + (w_3 \cdot \text{SentimentScore})$$
Where:
\begin{itemize}
    \item \textbf{HireRate}: Normalized value between 0 and 1.
    \item \textbf{SpendMetric}: Logarithmic scale of total dollars spent to prevent outlier bias.
    \item \textbf{SentimentScore}: A value derived from keyword matching in client reviews.
\end{itemize}

\subsection{Outputs}
\begin{itemize}
    \item \textbf{Visual Score:} A numerical value (1-100) displayed in the extension.
    \item \textbf{Traffic Light Status:} Red (Score $<$ 50), Yellow (50--79), Green ($\geq$ 80).
\end{itemize}

\section{FR-003: Freelancer Decision Support System}
\subsection{Description}
The DSS acts as the analytical "brain" of the extension. It synthesizes data to provide a final verdict on whether the user should spend "Connects" on a job.

\subsection{Inputs}
\begin{itemize}
    \item \textbf{Aggregated Client Score:} The 1-100 value from FR-002.
    \item \textbf{Job Recency:} Time since post.
    \item \textbf{Applicant Count:} Total current proposals submitted.
\end{itemize}

\subsection{Processing}
\begin{enumerate}
    \item \textbf{Tier 1 Logic:} If (Applicants $> 50$) AND (Hire Rate $< 30\%$), output "Skip."
    \item \textbf{Tier 2 Logic:} If (Score $> 80$) AND (Applicants $< 20$), output "Strong Apply."
    \item \textbf{Tier 3 Logic:} If (Post Time $< 30$ mins), add "Urgent Action" badge.
\end{enumerate}

\subsection{Outputs}
\begin{itemize}
    \item \textbf{Recommendation Badge:} Green (Apply), Yellow (Caution), Red (Skip).
    \item \textbf{Probability Estimate:} Percentage likelihood of the client viewing the proposal.
\end{itemize}

\section{FR-004: Proposal Template Management \& Ranking}
\subsection{Description}
Allows management of a library of templates. The system tracks which templates lead to interviews and ranks them by performance.

\subsection{Inputs}
\begin{itemize}
    \item \textbf{Template Body:} Text containing placeholders like [Client\_Name].
    \item \textbf{Performance Data:} User-marked success (Interview/Hire).
\end{itemize}

\subsection{Processing}
\begin{enumerate}
    \item \textbf{Success Velocity ($V$):} $V = (\text{Interviews} / \text{Total Uses}) \times 100$.
    \item \textbf{Auto-Ranking:} Reorder list based on $V$ value.
    \item \textbf{Parsing:} Replace brackets with data from FR-001.
\end{enumerate}

\subsection{Outputs}
\begin{itemize}
    \item \textbf{Ranked Library:} A searchable list sorted by performance.
    \item \textbf{Performance Icons:} "Top Performer" or "Needs Improvement."
\end{itemize}

\section{FR-005: Proposal History and Analytics}
\subsection{Description}
Maintains a record of all generated proposals for user auditing and performance tracking.

\subsection{Inputs}
\begin{itemize}
    \item \textbf{Proposal Text:} Final draft content.
    \item \textbf{Timestamp:} Date of generation.
\end{itemize}

\subsection{Processing}
\begin{enumerate}
    \item \textbf{Aggregation:} Weekly totals of applications submitted.
    \item \textbf{Trend Calculation:} Compare current week success rate to previous week.
\end{enumerate}

\subsection{Outputs}
\begin{itemize}
    \item \textbf{Dashboard:} Graphical bar and pie charts.
    \item \textbf{Export:} CSV/PDF generation.
\end{itemize}

\section{FR-006: User Profile and Skills Configuration}
\subsection{Description}
A knowledge base for the AI, storing the freelancer's specific background and technical strengths.

\subsection{Inputs}

\chapter{Non-Functional Requirements}

\section{Introduction}
While functional requirements define what the system does, this chapter defines the quality attributes, operational constraints, and performance benchmarks that ensure the system is professional-grade. In adherence to IEEE Std 830-1984, these requirements are quantifiable, measurable, and verifiable.

\section{Performance Requirements}
\subsection{Speed and Latency}
\begin{itemize}
    \item \textbf{NFR-P-01 (UI Responsiveness):} The extension popup and on-page overlays shall render and become interactive within \textbf{1.5 seconds} of the trigger event.
    \item \textbf{NFR-P-02 (Analysis Execution):} The Client Reliability Analysis (FR-001/FR-002) shall complete processing and display results within \textbf{3.0 seconds} of data extraction.
    \item \textbf{NFR-P-03 (Generation Latency):} AI-based proposal generation shall deliver a draft to the user in under \textbf{5.0 seconds}, excluding external network API delays.
\end{itemize}

\subsection{Resource Utilization}
\begin{itemize}
    \item \textbf{NFR-P-04 (Memory Management):} The browser extension shall not consume more than \textbf{180MB of RAM} at peak processing to ensure no degradation of the browser experience.
    \item \textbf{NFR-P-05 (CPU Overhead):} Background script CPU usage shall remain below \textbf{3\%} during idle states and \textbf{10\%} during active generation.
\end{itemize}

\section{Reliability and Availability}
\begin{itemize}
    \item \textbf{NFR-R-01 (Availability):} The system backend and template database shall maintain \textbf{99.9\% uptime} (Three Nines), measured on a monthly basis.
    \item \textbf{NFR-R-02 (Fault Tolerance):} In the event of an external AI API failure, the system shall provide a fallback to manual templates within \textbf{500ms} without crashing the UI.
    \item \textbf{NFR-R-03 (Data Persistence):} User drafts and history shall be automatically cached locally every \textbf{30 seconds} to prevent data loss during browser crashes.
\end{itemize}

\section{Security and Privacy}
\begin{itemize}
    \item \textbf{NFR-S-01 (Encryption):} All data in transit between the extension and the server shall use \textbf{TLS 1.3} encryption over HTTPS.
    \item \textbf{NFR-S-02 (Storage Security):} Sensitive user data, including API keys and credentials, shall be stored using \textbf{AES-256} encryption at rest.
    \item \textbf{NFR-S-03 (Anonymization):} During data extraction, no sensitive client personal identifiable information (PII) shall be stored on the system backend.
\end{itemize}

\section{Usability and Human Factors}
\begin{itemize}
    \item \textbf{NFR-U-01 (Learnability):} A new user shall be able to perform a "Client Analysis" and "Proposal Generation" within \textbf{2 minutes} of installation without reading a manual.
    \item \textbf{NFR-U-02 (Accessibility):} The user interface shall comply with \textbf{WCAG 2.1 Level AA} standards, including high-contrast support and screen-reader compatibility.
\end{itemize}
